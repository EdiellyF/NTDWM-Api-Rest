SLIDE 1

A Evolução do Vue.js

2014

Evan You cria o Vue após trabalhar no Google Creative Lab — buscando o melhor entre Angular e React

Vue 2 · 2016

Options API: data(), methods, computed, watch — organização por tipo

Vue 3 · 2020

Composition API: ref(), reactive(), onMounted() — organização por funcionalidade

O fim do this

// Options API (Vue 2) — this por toda parte
export default {
  data() { return { lista: [] } },
  methods: { carregar() { this.lista = [...] } }  // this.lista
}

// Composition API (Vue 3) — sem this
const listaMateriais = ref([])
async function carregarDados() {
  listaMateriais.value = resposta.data   // direto, sem this
}
Em 2014, Evan You olhou para o ecossistema e disse: eu quero a declaratividade do Angular sem o peso, e a componentização do React sem a complexidade. Nasceu o Vue. O Vue 2 nos trouxe a Options API — você organizava o código por "tipo": todos os dados em data(), todos os métodos em methods. Funcionava, mas quando a funcionalidade crescia, você pulava entre blocos diferentes. O Vue 3 veio com a Composition API e resolveu isso. No meu projeto, eu uso script setup em todos os componentes. No meu stores/materiais.js, eu declaro const listaMateriais = ref([]) e a função carregarDados() logo abaixo — tudo relacionado à carga de dados fica junto, num bloco lógico. E o this? Acabou. Quando eu chamo materiais.carregarDados() dentro do onMounted no meu ListaMateriais.vue, eu não uso this.carregarDados() — acesso diretamente a constante importada da store.

SLIDE 2

O Coração do Vue — Reatividade e Diretivas

ref() — Reatividade

// stores/materiais.js
const listaMateriais = ref([])
const carregando = ref(false)
const erro = ref(null)

v-for — Renderização de lista

<!-- ListaMateriais.vue -->
<div v-for="item in materiais.listaMateriais"
     :key="item.id">

v-model — Two-way binding

<!-- CadastroMateriais.vue -->
<input v-model="formData.name" />
<input v-model.number="formData.amountStored" />

v-if / v-else-if — Condicional

<!-- ListaMateriais.vue -->
<div v-if="materiais.carregando">Carregando...</div>
<div v-else-if="materiais.erro">{{ materiais.erro }}</div>
<div v-else-if="listaMateriais.length === 0">Vazio</div>
<div v-else class="material-grid">...</div>
ListaMateriais.vue
:nomeMaterial="item.name"
:quantidade="item.amountStored"
→ props →
MaterialCard.vue
defineProps({ nomeMaterial, quantidade })
A reatividade é o coração do Vue. Quando eu declaro const listaMateriais = ref([]) na minha store, o Vue cria um Proxy por baixo dos panos. Qualquer alteração em listaMateriais.value dispara uma atualização automática na interface. As diretivas tornam isso visível: v-for para iterar sobre materiais.listaMateriais, :key para identificação única. O v-model é two-way binding — no CadastroMateriais.vue, quando o usuário digita, formData.name atualiza automaticamente. O modificador .number converte para Number. O v-if / v-else-if é renderização condicional real — encadeio quatro estados: carregando, erro, lista vazia, lista com dados. E a comunicação entre componentes: no ListaMateriais.vue, eu passo dados para o filho MaterialCard via props — :nomeMaterial e :quantidade. O filho recebe com defineProps e renderiza. Dados descem via props, eventos sobem via emits.

Lazy loading com () => import()

📦 Pinia — stores/materiais.js

export const useMateriaisStore = defineStore('materiais', () => {
  const listaMateriais = ref([])
  async function carregarDados() { ... }
  async function criarMaterial()  { ... }
  async function deletarMaterial() { ... }
  async function atualizarMaterial() { ... }
  return { listaMateriais, carregando, erro,
           carregarDados, criarMaterial,
           deletarMaterial, atualizarMaterial }
})

A arquitetura tem três pilares: Vue Router, Pinia e Axios. O Vue Router é responsável pela navegação sem reload — configurei quatro rotas com lazy loading, cada view só é baixada quando o usuário acessa aquela rota. O Pinia é o gerenciador de estado global — escolhi Pinia em vez de Vuex porque ele foi construído para a Composition API. A store expõe listaMateriais, carregando, erro e quatro funções assíncronas. Qualquer componente que importar useMateriaisStore acessa o mesmo estado reativo. O Axios é a ponte com o microserviço — criei uma instância com baseURL apontando para /api, que o nginx faz proxy para o Spring Boot. Nas stores, consumo com async/await — o await pausa a execução até a Promise resolver, e o try/catch garante tratamento de erro.

SLIDE 4

O Mercado — Prós e Contras

✅ Pontos Fortes

  • Curva de aprendizado baixa — <script setup> é intuitivo
  • Flexibilidade — Options API ou Composition API
  • Reatividade granular — atualiza só o que mudou
  • Pinia > Vuex — estado global sem boilerplate
  • SFC — HTML + JS + CSS num só arquivo
  • Documentação excepcional (traduzida PT-BR)

⚠️ Pontos Fracos

  • Ecossistema corporativo menor que React/Angular
  • Bancos e grandes empresas ainda usam Angular/React
  • Menos vagas "enterprise" no mercado brasileiro
  • Adoção em grandes corporações ainda tímida
  • SSR/SEO exige Nuxt — camada extra de complexidade
  • TypeScript menos "first-class" que no React

"Vue é a escolha certa para produtividade e times ágeis. Para o mercado enterprise brasileiro, saiba React ou Angular também — mas o futuro do Vue é promissor, e a Composition API prova que ele evolui na direção certa."

Nenhum framework é perfeito. Os pontos fortes do Vue são inegáveis: a curva de aprendizado é a menor entre os três grandes. No meu projeto, escrevo script setup, declaro ref(), uso v-model e já tenho interface funcional. O Pinia eliminou o boilerplate do Vuex — sem mutations, sem commit e dispatch. Mas os pontos fracos existem: no mercado corporativo brasileiro, bancos e grandes empresas ainda têm projetos massivos em Angular e React. O ecossistema de bibliotecas enterprise é mais robusto no React. E se precisa de SSR para SEO, o Nuxt adiciona complexidade. A mensagem final: Vue é a escolha certa para produtividade, para times pequenos e médios. Para o mercado enterprise, é importante saber React ou Angular também. Mas o futuro do Vue é promissor.
